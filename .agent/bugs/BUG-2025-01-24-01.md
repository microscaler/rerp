## Summary

The `local_resource` shell commands in `create_microservice_lint`, `create_microservice_gen`, `bff-spec-gen`, and `bff-lint` ended with `echo "Ō£ģ ... complete"` statements. Without `set -e`, if the main commands failed (e.g., linting or code generation), the script continued and the final `echo` returned exit code 0, causing Tilt to mark the resource as successful. This silently masked failures.

## Discovery

**Date:** 2025-01-24  
**Source:** Tilt/dev workflow review  
**Severity:** `high`  
**Status:** `fixed`

## Location

- **File:** `Tiltfile`
- **Resources:** `create_microservice_lint`, `create_microservice_gen`, `bff-spec-gen`, `bff-lint` (helper `cmd` strings)

## Description

Each of these `local_resource` `cmd` blocks ran one or more commands (e.g. `brrtrouter-gen lint`, `brrtrouter-gen generate`, `bff-generator generate-spec`) and finished with `echo "Ō£ģ ... complete"`. The shell does not exit on the first failing command by default. When lint or codegen failed, the script still reached the `echo`, which exits 0, so Tilt treated the resource as OK and showed the success message even though the underlying operation had failed.

## Root Cause

The `cmd` strings did not enable `set -e` (exit immediately on command failure). A failing `brrtrouter-gen` or `bff-generator` invocation therefore did not abort the script; execution continued to the final `echo`, whose exit code 0 was reported to Tilt.

## Fix

Added `set -e` at the beginning of the `cmd` string for:

- `create_microservice_lint`
- `create_microservice_gen`
- `bff-spec-gen`
- `bff-lint`

If any command in the block fails, the shell now exits with that commandŌĆÖs nonŌĆæzero status, and Tilt correctly reports the resource as failed.
