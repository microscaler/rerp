# BUG-2025-01-24-02: create_microservice_build_resource Dead target_path/artifact_path and Inconsistent Path Logic

**Date:** 2025-01-24  
**Status:** ✅ FIXED  
**Priority:** Medium  
**Severity:** Maintainability—dead code and misleading inconsistency

## Summary

The `create_microservice_build_resource` function computed `target_path` and `artifact_path` (lines 269–270) but never used them in the `local_resource` call. The return value `(binary_name, artifact_path)` was discarded at all call sites (lines 393, 453). Additionally, `target_path` used `binary_name`, while `create_microservice_deployment` correctly uses `package_name` for the same path (binary on disk uses Cargo [package] name). This dead code and inconsistency could mislead future maintainers.

## Discovery

**Date:** 2025-01-24  
**Source:** Code review / user report  
**Severity:** `medium`  
**Status:** `fixed`

## Location

- **File:** `Tiltfile`
- **Lines:** 266–293 (create_microservice_build_resource)
- **Call sites:** 393, 453

## Description

- `target_path` was set to `microservices/target/.../debug/%s` % binary_name but never passed to `local_resource` (deps, cmd, etc.). The build is done by `build-microservice.sh`, which uses the Cargo [package] name for the output path; `create_microservice_deployment`’s copy step correctly uses `PACKAGE_NAMES` for `target_path`.
- `artifact_path` was set to `build_artifacts/%s` % binary_name but never used. `create_microservice_deployment` uses `build_artifacts/amd64/%s` for the copy destination, so the value in `create_microservice_build_resource` was both unused and wrong.
- The function returned `(binary_name, artifact_path)`; both call sites call `create_microservice_build_resource(name)` without using the return value.

## Root Cause

Leftover or speculative variables and return value that were never wired into the `local_resource` or callers. The path logic was also inconsistent with `create_microservice_deployment`, which correctly derives the binary path from `PACKAGE_NAMES` and the artifact path from `build_artifacts/amd64/` and `BINARY_NAMES`.

## Fix

- Removed unused `target_path`, `artifact_path`, and `binary_name` from `create_microservice_build_resource`.
- Removed the `return (binary_name, artifact_path)`; call sites did not use it.
- Added a short comment that the build script emits to `microservices/target/.../debug/<package_name>` and that `create_microservice_deployment`’s copy step uses `PACKAGE_NAMES` for that path and `BINARY_NAMES` for `build_artifacts/amd64/`.

## Testing

- Tiltfile is Starlark; no automated test. Manual: run `tilt up` and ensure `build-<name>` resources still build and that `copy-<name>`, `docker-<name>`, and deployments behave as before.

## Impact

- **Before:** Dead code and a `target_path` using `binary_name` instead of `package_name` could lead maintainers to reuse those variables or assume path logic was shared. The unused return suggested the function might need to be used differently.
- **After:** No dead code; a single comment directs maintainers to `create_microservice_deployment` for binary and artifact path semantics.
