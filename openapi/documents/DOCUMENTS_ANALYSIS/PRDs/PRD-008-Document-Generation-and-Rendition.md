# PRD-008: Document Generation and Rendition

- **Status**: Proposed
- **Owner**: Documents / Render
- **First consumer**: Accounting customer invoices and credit notes through Hauliage
- **Related roadmap**: Accounting Goal 6 — Public Invoice-to-GL vertical slice
- **Target**: MVP rich-document rendering, followed by post-MVP trust services
- **Last updated**: 2026-07-15

## 1. Executive Summary

RERP needs one suite-level capability for generating professional documents from immutable, typed business facts. Accounting invoices and credit notes are the first consumer, but the same capability will later serve quotes, statements, remittances, contracts, HR letters, reports, and other suite-owned documents. The delivered Accounting Phase 1 renderer proves artifact storage and retrieval, but its layout is hard-coded in Rust and is a transitional vertical slice rather than the permanent ownership boundary.

The Documents `render` component will own an externalized, versioned HTML/CSS template system, rendering operations, generated artifacts, copies, storage linkage, and later trust services. Source suites own their business lifecycle and immutable render facts. When Accounting posts an invoice, it freezes the complete invoice render model and submits it to Documents after commit. Documents freezes the selected template version when it accepts the idempotent render operation. Mutable customer, legal-entity, or template data must never change an issued invoice.

The MVP will also distinguish an original artifact from an explicitly generated copy. A copy is a new immutable artifact marked `COPY` with its own generation time and lineage; ordinary retrieval continues to return the original bytes.

Cryptographic electronic sealing, qualified timestamps, PAdES long-term validation, and “notarised-style” visible trust panels are important but are post-MVP capabilities. The MVP architecture must leave a clean pre-storage signing boundary so these services can be added without redesigning templates or breaking artifact immutability.

## 2. Product Outcome

An ordinary RERP suite can submit a typed render model and receive an immutable generated document without embedding a PDF engine or template repository. For the first vertical slice, Hauliage can:

1. submit the commercial facts needed to issue an invoice;
2. receive an immutable posted accounting document and balanced journal;
3. retrieve a professionally rendered PDF using the issuer’s published template;
4. retry posting or rendering without creating another invoice or another original artifact; and
5. request a clearly identified copy without mutating or replacing the original.

An RERP administrator can create, preview, validate, publish, retire, and select templates without rebuilding Accounting or any other consuming suite.

## 3. Current State

The Accounting Phase 1 runtime already provides a transitional implementation whose proven behavior this Documents component must preserve:

- `GET /v1/customer-invoices/{id}/document` lazily materializes or retrieves an artifact;
- the renderer consumes a stored invoice snapshot rather than performing mutable customer or organization lookups;
- generated bytes are hashed and stored under a content-addressed MinIO key;
- artifact metadata records renderer name, renderer version, checksum, size, render actor, and render time; and
- callers receive a short-lived presigned download URL rather than public bucket access.

The current `rerp-basic-pdf` implementation is intentionally narrow. It uses a fixed Helvetica text layout, substitutes unsupported characters, exposes customer IDs rather than complete billing identities, and has no external template or asset contract. It remains in Accounting only until the Documents render runtime and Accounting handoff are accepted end to end; no rich rendering capability should be added to Accounting.

## 4. Decisions

### 4.0 Documents owns generation; source suites own meaning

Documents owns reusable presentation infrastructure: templates, assets, rendering, generated-document records, binaries, copies, downloads, retention metadata, watermarks, and future electronic sealing. A source suite owns the authoritative business record, its lifecycle and corrections, the typed render-model schema, and the decision to issue a document.

Documents must not query mutable source-suite master data or reinterpret domain facts. The consuming suite submits a frozen render model through an idempotent post-commit operation. Generated documents link back to the source using suite, resource type, resource ID, source version and snapshot checksum.

### 4.1 HTML and print CSS are the document presentation contract

Templates will use semantic HTML and a controlled print-CSS profile. This gives template authors familiar layout tools for tables, page breaks, headers, footers, logos, typography, and localization while keeping accounting calculations outside the template.

The renderer implementation is replaceable. The template contract must not expose browser-specific APIs or depend on JavaScript.

### 4.2 MVP uses one Jinja-compatible expression language

The initial expression language should be a constrained Jinja-compatible dialect implemented in Rust, with MiniJinja as the leading implementation candidate. It provides field interpolation, loops, conditionals, includes, macros, escaping, and controlled filters without introducing Python at runtime.

The MVP must not offer both Jinja and Handlebars. Supporting two expression languages would double validation, documentation, escaping, compatibility, and security obligations. Each template version records its engine contract, for example `minijinja/v1`, so another engine can be added later without ambiguity.

### 4.3 Templates contain presentation logic, not accounting logic

Templates may select, iterate, label, and format supplied values. They must not:

- execute SQL or arbitrary code;
- call RERP, Hauliage, Sesame, or internet endpoints;
- calculate authoritative tax, discounts, balances, or journal values;
- read the local filesystem or environment;
- mutate the render model; or
- load undeclared remote assets.

All authoritative monetary and tax facts come from the posted accounting snapshot.

### 4.4 Original invoices are immutable; rendering is idempotent materialization

`POSTED` is the immutability sentinel. Posting freezes the accounting facts, presentation facts, render-schema version, and template-version selection. The first successful original render becomes the only original PDF artifact for that invoice and format.

Re-requesting the document returns the stored original. A retry after a transient failure may repeat rendering, but equivalent frozen inputs under the same pinned renderer must produce the same unsigned bytes and checksum.

### 4.5 Copy creation is explicit

Downloading an invoice is not regeneration. Copy creation is a separate command that creates a new immutable child artifact with a visible, configurable copy mark. A copy never replaces the original and never changes accounting state.

### 4.6 Qualified trust services are post-MVP

A visible seal, stamp, or signature image is not cryptographic evidence and must never be described as notarisation by itself. Post-MVP trust services will add electronic seals, trusted timestamps, validation evidence, and optional visible trust panels through a provider-neutral boundary.

## 5. Scope

### 5.1 MVP scope

- Customer invoice and customer credit-note PDFs.
- Complete immutable render-data snapshots.
- Versioned, tenant- and legal-entity-scoped templates.
- HTML, print CSS, approved fonts, logos, and declared static assets.
- Template validation, preview, publication, retirement, and selection.
- A default RERP template usable without tenant configuration.
- Full Unicode text and locale-aware formatting.
- Multi-page tables, repeating headers/footers, and controlled page breaks.
- Immutable original artifacts in the existing private object store.
- Explicit, idempotent copy creation with visible copy metadata.
- RLS enforcement, audit events, metrics, and structured failures.

### 5.2 Post-MVP scope

- Qualified or advanced electronic seals for issuing legal entities.
- Qualified electronic timestamps.
- PAdES Baseline `T`, `LT`, and optionally `LTA` profiles.
- Qualified trust-service-provider integrations, including InfoNotary or equivalent providers.
- Signature/seal validation and preservation workflows.
- Visible trust panels and validation QR codes.
- Jurisdiction-specific trust and archival policies.
- A visual/WYSIWYG template designer.
- Additional accounting documents such as statements, payment notices, remittances, and vendor documents.
- PDF/A and PDF/UA certification where required.
- E-invoice containers such as Factur-X/ZUGFeRD or country-specific formats.

### 5.3 Explicitly out of scope for MVP

- Template-supplied JavaScript.
- Arbitrary executable plugins, shell commands, SQL, or network calls.
- Recalculating accounting totals in a template.
- Mutating or replacing a posted invoice.
- Treating a visual stamp as a qualified signature, seal, timestamp, or notarial act.
- Selecting templates from untrusted URLs at render time.
- Email delivery and delivery-receipt workflows.

## 6. Users and Use Cases

### 6.1 Accounting operator

- Issues an invoice and obtains the immutable original PDF.
- Retrieves the same original later.
- Creates a marked copy when a customer requests a regenerated document.
- Sees whether rendering is pending, ready, or failed without losing the posted invoice.

### 6.2 Tenant or legal-entity administrator

- Uploads a template bundle and assets.
- Validates and previews it against synthetic or authorized draft data.
- Publishes an immutable version for a document type and locale.
- Selects an effective template without modifying prior invoices.
- Retires a template so it cannot be selected for future invoices.

### 6.3 API consumer

- Supplies commercial and billing facts through a versioned invoice contract.
- Does not need to know RERP’s template engine or object-store layout.
- Retries safely using idempotency keys.
- Receives stable artifact metadata and time-limited download access.

### 6.4 Auditor or support operator

- Determines which facts, template, renderer, and actor produced an artifact.
- Verifies its checksum and lineage.
- Distinguishes the original from every copy.
- Confirms that mutable master-data changes did not alter an issued document.

## 7. Lifecycle and Invariants

### 7.1 Source business-document lifecycle

```text
DRAFT --post--> POSTED
```

- Drafts may change under normal authorization and validation rules.
- Posting is a one-way transition.
- Posted Accounting facts and Accounting-owned render-snapshot rows cannot be updated or deleted through application APIs.
- Database constraints or triggers must enforce the invariant in addition to application checks.
- Corrections use credit notes and new invoices, never edits to posted facts.

### 7.2 Template lifecycle

```text
DRAFT --publish--> PUBLISHED --retire--> RETIRED
```

- A draft version may be edited and previewed.
- Publication freezes template body, CSS, manifest, assets, schema compatibility, and checksum.
- A published version is immutable.
- Retirement prevents future selection but does not affect documents that reference the version.
- Editing a published design creates another version.

### 7.3 Artifact lifecycle

```text
PENDING --render--> READY
   |                  |
   +----failure----> FAILED --retry--> READY
```

- Source-suite persistence remains atomic and does not depend on Documents, object-store or renderer availability.
- Artifact generation may complete after posting.
- Exactly one `ORIGINAL` PDF is permitted for the frozen invoice rendition.
- Every `COPY` is a separate immutable artifact linked to the original.
- `READY` artifact bytes and metadata are append-only and content-addressed.
- Repair of missing or corrupt storage is an explicit audited recovery operation, not ordinary regeneration.

## 8. Immutable Render Model

Documents must render from a typed, versioned model submitted by the owning suite rather than dynamically joining mutable customer or company records. For invoices, the render model is presentation-ready but remains derived from authoritative Accounting facts.

### 8.1 Required render-model groups

| Group | Required content |
|---|---|
| Document | ID, number, type, issue date, due date, posting time, currency, locale, source reference |
| Issuer | Legal name, registered and trading addresses, company identifiers, tax registrations, contact details |
| Customer | Billing name, billing address, customer reference, company identifiers, tax registrations, contact details |
| Lines | Stable order, description, quantity, unit, unit price, discounts, tax classification, tax rate, tax amount, net and gross totals |
| Totals | Subtotal, discounts, tax summaries, rounding, total, paid amount where applicable, amount due |
| Payment | Terms, due-date wording, supported payment instructions, bank-account display data and payment reference |
| Compliance | Jurisdiction-specific labels, exemptions, reverse-charge text, statutory notes and invoice-copy terminology |
| Presentation | Locale, timezone, page size, template version, branding selection, display preferences |
| Audit | Render-schema version, snapshot checksum, created time and source-document lineage |

Secrets and internal accounting information—tenant IDs, internal account IDs, object-store credentials, RLS context, or private notes—must not be exposed to templates.

### 8.2 Snapshot rules

- Billing and issuer facts are resolved and validated before or during posting.
- RERP stores the exact representation used for issuance.
- The snapshot is canonicalized and checksummed.
- Templates receive only fields declared by the render-model schema.
- Adding optional fields is backward compatible; removing or changing semantics requires another schema version.
- Template validation proves compatibility with its declared render-schema range before publication.

## 9. Template Bundle Contract

A template version is uploaded as a bounded bundle containing:

```text
manifest.json
invoice.html
invoice.css
partials/
assets/
```

An illustrative manifest is:

```json
{
  "template_contract": "rerp-invoice-template/v1",
  "render_schema": "invoice-render-model/v1",
  "engine": "minijinja/v1",
  "entrypoint": "invoice.html",
  "stylesheets": ["invoice.css"],
  "document_types": ["CUSTOMER_INVOICE", "CUSTOMER_CREDIT_NOTE"],
  "locales": ["en", "bg"],
  "page": { "size": "A4", "orientation": "portrait" }
}
```

An illustrative template fragment is:

```html
<h1>{{ labels.invoice }}</h1>
<p class="invoice-number">{{ document.number }}</p>

<tbody>
{% for line in lines %}
  <tr>
    <td>{{ line.description }}</td>
    <td class="number">{{ line.quantity | decimal }}</td>
    <td class="money">{{ line.total | money(currency=document.currency) }}</td>
  </tr>
{% endfor %}
</tbody>
```

### 9.1 Supported capabilities

- Auto-escaped field interpolation.
- Bounded loops over supplied collections.
- Simple conditionals.
- Declared partials/includes inside the bundle.
- Macros with resource limits.
- RERP-owned formatting filters for money, decimals, dates, identifiers, addresses, and translated labels.
- CSS print layout, embedded approved fonts, local images, tables, page breaks, headers, and footers.

### 9.2 Prohibited capabilities

- Raw unescaped HTML by default.
- Dynamic include paths.
- Filesystem, environment, process, SQL, HTTP, or DNS access.
- JavaScript or browser extensions.
- Data-dependent unbounded recursion.
- External font, image, stylesheet, iframe, or URL fetches.
- Template-defined authoritative arithmetic.

### 9.3 Validation and publication

Validation must check:

- archive and uncompressed size limits;
- allowed file count, path structure, MIME types, and asset dimensions;
- manifest schema and engine compatibility;
- syntax and undeclared-field references;
- missing partials and assets;
- prohibited HTML, CSS, URI, and template constructs;
- required output semantics such as document number, issuer, customer, dates, lines, currency, tax, and total;
- successful rendering against boundary fixtures; and
- renderer time, memory, page-count, and output-size limits.

Publication stores the validated bundle checksum and creates an audit event.

## 10. Template Selection

RERP selects templates using trusted configuration rather than accepting an arbitrary template body with an invoice command.

Selection inputs, in order of specificity, are:

1. legal entity;
2. accounting document type;
3. locale and jurisdiction;
4. effective date; and
5. tenant default, followed by the built-in RERP fallback.

The resolved template-version ID is written into the posting/render snapshot. Changes to selection policy affect only later postings.

## 11. Functional Requirements

### 11.1 Posting and immutability

- **FR-PDF-001**: RERP shall make `POSTED` a one-way, database-enforced state for accounting and render facts.
- **FR-PDF-002**: Posting shall freeze a canonical render model, render-schema version, selected template-version ID, and their checksums.
- **FR-PDF-003**: Posted invoices shall be corrected only through credit-note or replacement-invoice workflows.
- **FR-PDF-004**: PDF renderer or object-store failure shall not partially roll back or duplicate a successfully posted invoice and journal.

### 11.2 Template administration

- **FR-PDF-010**: Authorized administrators shall create templates and mutable draft versions.
- **FR-PDF-011**: Administrators shall upload bounded HTML/CSS template bundles and assets through an authenticated API.
- **FR-PDF-012**: RERP shall validate syntax, schema compatibility, security policy, mandatory semantics, and boundary-fixture rendering before publication.
- **FR-PDF-013**: Administrators shall preview draft templates without creating an accounting artifact or changing invoice state.
- **FR-PDF-014**: Publishing shall make the version immutable and auditable.
- **FR-PDF-015**: Retiring a template shall prevent future selection without affecting existing documents.
- **FR-PDF-016**: RERP shall always provide a compatible default template.

### 11.3 Original rendering

- **FR-PDF-020**: RERP shall render invoice and credit-note PDFs from frozen snapshots only.
- **FR-PDF-021**: The original PDF shall display issuer, customer, document, line, tax, total, payment, and required compliance facts supplied by the render model.
- **FR-PDF-022**: The renderer shall support Unicode, approved embedded fonts, logos, multi-page line tables, repeating page elements, and A4/Letter configuration.
- **FR-PDF-023**: The same frozen model, published template, assets, and pinned renderer shall produce the same canonical unsigned PDF bytes.
- **FR-PDF-024**: Only one original artifact shall become `READY` for one invoice rendition and media type.
- **FR-PDF-025**: Repeated retrieval shall return the existing original artifact and must not rerender it.
- **FR-PDF-026**: Original artifact metadata shall identify the model, template, renderer, checksums, actor, creation time, storage location, and media type.
- **FR-PDF-027**: RERP shall provide a short-lived authorized download URL without making the object public.

### 11.4 Copy rendering

- **FR-PDF-030**: Copy creation shall be a separate authenticated command with its own idempotency key.
- **FR-PDF-031**: A copy shall retain the original accounting content, template version, and visual design.
- **FR-PDF-032**: A copy shall display a configurable mark such as `COPY` or `DUPLICATE`, copy-generation time, and original document number.
- **FR-PDF-033**: Copy terminology shall be selected by jurisdiction and document policy, not arbitrary caller text.
- **FR-PDF-034**: A copy shall have its own artifact ID, checksum, object key, generation actor, reason, and `source_artifact_id` lineage.
- **FR-PDF-035**: Retrying a copy command with the same idempotency key and payload shall return the same copy; changing the payload shall return a conflict.
- **FR-PDF-036**: A copy shall never replace or modify the original artifact.

### 11.5 Retrieval and audit

- **FR-PDF-040**: APIs shall distinguish `PENDING`, `READY`, and `FAILED` artifact generation states.
- **FR-PDF-041**: Authorized callers shall retrieve original metadata and enumerate copies without learning another tenant’s artifacts.
- **FR-PDF-042**: Every template transition, render attempt, successful artifact, copy command, recovery action, and future signing operation shall create an audit event.
- **FR-PDF-043**: Failures shall expose stable machine-readable error codes and correlation IDs without leaking template data, filesystem paths, credentials, or tenant details.

## 12. Proposed API Surface

Exact names remain OpenAPI-first design work, but the product capabilities should map to resources similar to:

```text
POST   /v1/document-templates
GET    /v1/document-templates/{template_id}
POST   /v1/document-templates/{template_id}/versions
POST   /v1/document-template-versions/{version_id}/validate
POST   /v1/document-template-versions/{version_id}/preview
POST   /v1/document-template-versions/{version_id}/publish
```

## 13. Proposed Persistence Model

Names are illustrative and must be finalized through Lifeguard entities and migrations.

| Record | Responsibility |
|---|---|
| `document_templates` | Stable tenant/legal-entity template identity, purpose and ownership |
| `document_template_versions` | Immutable published manifest, engine, render-schema compatibility, body and checksum |
| `document_template_assets` | Content-addressed fonts, logos, styles and other approved bundle assets |
| `document_template_policies` | Effective template selection by source type, legal entity, locale, jurisdiction and date |
| `document_render_operations` | Accepted immutable render model, source lineage, snapshot checksum, template selection and idempotent operation state |
| `document_renditions` | Original/copy metadata, object location, checksum, renderer lineage and trust status |
| `document_rendition_operations` | Copy/signing attempt idempotency and stable result/error information |

Documents shall permit exactly one `ORIGINAL` for a source resource, source version, rendition purpose and media type, plus multiple independently identified `COPY` artifacts. The transitional Accounting artifact constraint will be retired at cutover rather than becoming the Documents model.

## 14. Rendering Architecture

```text
Source-suite transaction
  ├─ persist immutable business facts
  ├─ persist source-owned render snapshot
  ├─ persist document-generation outbox operation
  └─ commit
          ↓
Post-commit Documents render operation
  ├─ load snapshot and published bundle under RLS
  ├─ freeze effective template version against source issue time
  ├─ evaluate constrained Jinja-compatible template
  ├─ render HTML/CSS with a pinned renderer and font set
  ├─ normalize deterministic PDF metadata
  ├─ validate PDF structure and required semantics
  ├─ hash and upload immutable bytes
  └─ persist READY artifact metadata
```

Renderer implementation must be isolated from request-handling workers sufficiently to enforce CPU, memory, execution-time, filesystem, and network limits. Whether this is a supervised process or dedicated worker is an implementation ADR decision; the public ownership boundary remains the Documents `render` component and must not bypass Sesame/RLS controls.

All outbound calls—including a future trust-service provider—must use the Microscaler rustls/may-compatible HTTP stack. Templates themselves never make outbound calls.

## 15. Non-Functional Requirements

### 15.1 Security and isolation

- **NFR-PDF-001**: Template, snapshot, artifact, and policy records shall be protected by forced PostgreSQL RLS.
- **NFR-PDF-002**: The rendering sandbox shall have no general network access and only bounded read access to materialized bundle inputs.
- **NFR-PDF-003**: JavaScript, external URIs, dynamic filesystem access, and arbitrary code execution shall be disabled.
- **NFR-PDF-004**: Upload validation shall defend against path traversal, archive bombs, decompression bombs, malformed fonts/images, excessive dimensions, recursive templates, and resource exhaustion.
- **NFR-PDF-005**: Templates shall receive an allowlisted view model that excludes internal and cross-tenant data.
- **NFR-PDF-006**: Object-store credentials and future signing keys shall never appear in templates, snapshots, logs, API responses, or artifact object keys.

### 15.2 Correctness and reproducibility

- **NFR-PDF-010**: Displayed totals and tax shall match persisted decimal accounting facts exactly; templates cannot recalculate them.
- **NFR-PDF-011**: The renderer binary/container, browser or PDF engine, fonts, locale data, template engine, and normalization rules shall all have pinned versions.
- **NFR-PDF-012**: Given the same frozen inputs and pinned runtime, canonical unsigned rendering shall produce the same SHA-256 checksum across retries and replicas.
- **NFR-PDF-013**: Rendering and copy commands shall be safe under concurrent duplicate requests.

### 15.3 Performance and capacity

- **NFR-PDF-020**: A normal invoice of up to 100 lines shall render within 5 seconds at p95 on the supported shared-cluster profile, excluding any post-MVP external trust-service latency.
- **NFR-PDF-021**: A 500-line boundary invoice shall render within 15 seconds or fail with an explicit resource-limit error rather than destabilize the service.
- **NFR-PDF-022**: MVP shall enforce configurable bundle, asset, page-count, and PDF-size limits with documented defaults.
- **NFR-PDF-023**: Rendering shall not hold a database transaction or Lifeguard pool slot during template evaluation, PDF generation, object upload, or external signing.

### 15.4 Availability and recovery

- **NFR-PDF-030**: Renderer or object-store unavailability shall leave the posted invoice valid and the artifact operation retryable.
- **NFR-PDF-031**: Operation state shall survive process and pod restarts.
- **NFR-PDF-032**: Recovery shall detect an existing content-addressed object and reconcile metadata without duplicating the original.
- **NFR-PDF-033**: Missing original bytes after a recorded successful upload shall raise an integrity incident; ordinary retrieval shall not silently rerender history.

### 15.5 Observability

- **NFR-PDF-040**: Metrics shall cover queued, active, successful, failed, retried and timed-out renders; render duration; page count; output size; and template/renderer versions.
- **NFR-PDF-041**: Logs and traces shall correlate invoice, artifact and operation IDs while excluding invoice body data and credentials.
- **NFR-PDF-042**: Alerts shall distinguish template defects, renderer capacity, object-store failures, and future trust-provider failures.

### 15.6 Internationalization and quality

- **NFR-PDF-050**: MVP shall preserve Unicode text using embedded, licensed fonts and shall not replace valid characters with `?`.
- **NFR-PDF-051**: Dates, decimals, currency, addresses and labels shall use the frozen document locale and timezone rules.
- **NFR-PDF-052**: Templates shall use semantic HTML and maintain logical reading order so later PDF/UA support does not require a complete redesign.
- **NFR-PDF-053**: Generated PDFs shall open without warnings in supported PDF validators and common desktop/browser viewers.

## 16. MVP Acceptance Criteria

### 16.1 Immutable original

Given a valid invoice and published template, when the invoice is posted and its document becomes ready, then:

- one original PDF artifact exists;
- the PDF shows the same number, lines, tax and totals as the posted accounting document;
- the artifact references the frozen snapshot, template version and renderer version;
- repeated retrieval returns the same artifact ID and checksum; and
- later changes to customer, legal-entity, template-policy or template-draft data do not change it.

### 16.2 Idempotent concurrent rendering

Given no ready original and two concurrent retrieval/render attempts, both callers eventually observe the same original artifact. At most one original metadata record is committed, and any duplicate upload is the same content-addressed object.

### 16.3 Published-template immutability

Given a published template version, update and delete attempts fail. Creating and publishing another version changes only invoices posted after the new policy takes effect.

### 16.4 Secure template execution

Templates attempting external URL access, JavaScript execution, path traversal, undeclared includes, excessive recursion, or prohibited raw output fail validation or rendering with stable errors and cannot access the network, filesystem, environment, secrets, or another tenant’s data.

### 16.5 Unicode and pagination

A fixture containing Bulgarian and Latin text, long addresses, long descriptions, tax summaries, and more than one page renders without character substitution, clipped totals, lost table headers, or overlapping footer content.

### 16.6 Explicit copy

Given a ready original, when an authorized caller requests a copy with an idempotency key, then a new immutable artifact is created with:

- an appropriate visible copy mark;
- the copy-generation time;
- original document identification;
- original-artifact lineage; and
- its own checksum and object key.

Retrying the same command returns that artifact. Retrieving the original still returns its unchanged bytes.

### 16.7 Failure and retry

If rendering or object storage fails after invoice posting, the invoice and journal remain posted exactly once. The operation is visible as failed/retryable, and a later retry produces one original without another invoice or journal.

### 16.8 Tenant isolation

Cross-tenant template, snapshot, artifact, preview, copy and download attempts fail closed through the same Sesame and Lifeguard RLS boundary used by accounting data.

## 17. Post-MVP Trust Services and “Notarised-Style” Documents

### 17.1 Terminology

RERP must distinguish:

- **Visible stamp**: human-readable artwork or text; no cryptographic assurance by itself.
- **Electronic signature**: associated with a natural person and intent to sign.
- **Electronic seal**: associated with a legal entity and suitable for automated proof of origin and integrity.
- **Electronic timestamp**: binds trusted time to document data.
- **Notarial act**: a regulated legal process that RERP must not claim merely because a PDF is visibly stamped or digitally sealed.

For automatically issued invoices, an organizational electronic seal is the likely default. A human qualified electronic signature should be used only where policy or law requires a person to sign.

### 17.2 Target capability

The preferred long-term output is a PAdES document with:

- the issuing legal entity’s advanced or qualified electronic seal;
- a trusted or qualified timestamp;
- embedded certificate-chain and revocation evidence for long-term validation;
- an optional visible trust panel showing seal holder, timestamp, provider, validation status and a safe validation link/QR code; and
- preservation metadata suitable for the configured retention period.

PAdES Baseline `LT` is the likely operational target, with `LTA` used when archival policy requires periodic preservation timestamps. Final profiles require a regulatory and provider ADR.

### 17.3 Provider boundary

RERP shall define a provider-neutral trust-service adapter. InfoNotary is a valid candidate because its published services include automated qualified signing/sealing, qualified timestamps, validation, and PAdES profiles. RERP must not claim qualified status unless the selected provider, certificate, key custody, signing device, policy, and validation result meet the applicable requirements.

Private seal/signature keys must remain in an approved HSM, qualified signing device, or qualified provider infrastructure. They must not be stored in RERP configuration or Kubernetes secrets as exportable private keys.

### 17.4 Trust-aware artifact pipeline

```text
Canonical unsigned rendition
    ↓
Apply ORIGINAL/COPY visible presentation
    ↓
Electronic seal
    ↓
Trusted timestamp and validation material
    ↓
Validate final PAdES
    ↓
Hash and store final immutable bytes
```

The artifact checksum is calculated over the final sealed bytes. Because signature values and timestamps vary, signing idempotency cannot rely on regenerating identical signatures. RERP must persist a durable provider operation ID and return the already completed artifact on retry.

A copy mark must be applied before that copy is sealed. Adding an overlay to a final signed original may invalidate or alter its signature state and is prohibited.

### 17.5 Post-MVP acceptance gates

- A supported independent validator confirms the configured PAdES profile and complete certificate chain.
- Modifying any signed byte causes validation failure.
- The seal identifies the configured issuing legal entity.
- The trusted timestamp is not derived from the RERP application clock.
- Revocation and validation evidence remain available under the retention policy.
- Signing-provider retries cannot create competing originals.
- Provider outage leaves the artifact operation pending or retryable without duplicating accounting state.
- A copy is visibly marked before receiving its own seal and timestamp.
- The visible panel never claims “notarised” or “qualified” unless validated policy supports that claim.

## 19. Success Measures

- Every posted Hauliage invoice has one immutable, retrievable original PDF.
- A tenant can publish a branded template without deploying application code.
- No production invoice renderer performs mutable customer or organization lookups.
- Duplicate render and copy commands produce no duplicate accounting result or competing original.
- Cross-tenant document and template access tests fail closed.
- Template validation blocks all security-policy fixtures.
- Unicode and multi-page rendering pass golden-document and structural tests.
- Render failures are observable and recoverable without database transaction leakage.
- Post-MVP: independently validated sealed artifacts meet the configured PAdES and trust-policy profile.

## 20. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| HTML renderer expands the attack surface | No JavaScript/network, strict bundle validation, sandboxing, resource limits and pinned runtime |
| Templates become accounting programs | Typed presentation model, no authoritative arithmetic, controlled filters and mandatory semantic validation |
| Historical documents drift | Freeze model/template/renderer lineage and always retrieve stored original bytes |
| Renderer output differs across replicas | Pin binary, fonts and locale data; normalize metadata; checksum acceptance tests |
| A template breaks production issuance | Preview/boundary fixtures, publish gate, default fallback before posting, explicit artifact failure after posting |
| Copy stamping changes a signed original | Create copy presentation before sealing; never alter final original bytes |
| “Notarised” branding overstates legal status | Controlled vocabulary, validation-derived status and compliance approval |
| Trust-provider lock-in | Provider-neutral adapter and standard PAdES output |
| Trust-provider retries create several signatures | Durable provider operation idempotency and stored final result |

## 21. Open Decisions

The following require ADRs or implementation spikes before Slice 3:

1. Which pinned HTML-to-PDF engine best meets print CSS, deterministic output, sandboxing, Unicode, PDF structure, container size and licensing requirements?
2. Is the renderer a supervised child process, dedicated worker deployment, or narrow internal service?
3. Which exact CSS profile, font families, bundle limits, page limits and output limits are supported in MVP?
4. Is PDF/A required for every original or only selected jurisdiction/retention policies?
5. Which fields are mandatory in render-model v1 for the first Hauliage jurisdiction?
6. Is copy generation needed in the initial Hauliage UI, or only through the public API?
7. Which PAdES profile, trust-service provider and key-custody model will be selected post-MVP?
